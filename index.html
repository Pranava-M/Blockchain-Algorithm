Genesis Block — Introduction

Consensus Algorithms in Blockchain
— A Comparative Explorer.

Every blockchain is a room full of strangers who must agree on one shared truth without a referee. This page walks through how they agree, what it costs them, and which real networks made each trade-off — click an algorithm below and every panel updates together.

Block 01 · What is consensus?

Agreement without a referee

In a normal database, one company owns the machine and its word is final. A blockchain has no owner — thousands of independent computers ("nodes") hold copies of the same ledger. A consensus algorithm is the rulebook they all follow to agree on which transactions happened, in which order, without trusting each other or any central authority.

Definition

Consensus is the process by which a distributed network of independent nodes, none of whom trust each other, arrives at a single, agreed-upon version of the ledger — and stays protected from anyone rewriting history or spending the same coin twice (the "double-spend problem").

Generic consensus flow

01User broadcasts a transaction to the network
→
02Nodes validate it against the rules (signatures, balances)
→
03A node/validator is selected to propose the next block
→
04Other nodes verify and vote / re-compute proof
→
05Block is appended; ledger copies sync across all nodes
Key points
  • Immutability: once a block is buried under enough later blocks, rewriting it would mean redoing all the work/stake after it — practically impossible, so history becomes permanent.
  • Transparency: every node holds the full ledger, so anyone can independently audit every transaction ever made.
  • Decentralisation: no single party can unilaterally approve or block a transaction — power is spread across many independent participants.
  • Fault tolerance: the network keeps working correctly even if some nodes are offline, faulty, or actively malicious.
  • Different algorithms choose different mixes of speed, cost, and openness — there is no universally "best" one, only trade-offs suited to different goals.

Block 03 · The Blockchain Trilemma

Pick two, and negotiate the third

Every consensus algorithm is judged on three pillars. Improving one tends to strain the other two — this tension is called the Blockchain Trilemma.

ScalabilityHow many transactions the network can process, measured in TPS (transactions per second) and how fast a new block is created ("block time"). Higher TPS + shorter block time = more scalable.
SecurityResistance to attacks — especially a 51% attack (one party controlling most of the network's power) and Sybil attacks (one entity faking many identities to gain influence).
DecentralisationHow widely spread control is — the number of independent nodes/validators needed to participate, and how cheap/accessible it is to become one.
Key points
  • Bitcoin-style Proof of Work maximises security and decentralisation but sacrifices scalability (≈10 min blocks).
  • Delegated models (DPoS) maximise scalability by shrinking the validator set — but that concentrates power, hurting decentralisation.
  • Layer 2 networks exist specifically to route around this trilemma: they inherit Layer 1's security while processing transactions off-chain for speed.
Block 04 · Security attacks

What consensus is actually defending against

"Security" in the trilemma isn't abstract — it means resisting a specific list of known attack patterns. Every algorithm above is a different answer to the same question: how expensive can we make each of these?

51% Attack

An attacker who controls the majority of the network's mining power (PoW) or staked coins (PoS) can rewrite recent history and double-spend coins by out-producing the honest chain.

Mitigation: make majority control prohibitively expensive — via real hardware/energy cost (PoW) or slashed capital (PoS).
Sybil Attack

An attacker creates a huge number of fake node identities to gain disproportionate voting or influence power over the network, since a permissionless network can't verify "one node = one real entity."

Mitigation: tie influence to a scarce resource (computation, stake, or verified identity) that can't be freely duplicated.
Eclipse Attack

An attacker surrounds a single victim node with malicious peers, controlling every connection it has — feeding it a false view of the network to enable double-spends or wasted mining effort.

Mitigation: diverse, randomised peer selection and a minimum number of outbound connections to trusted-ish sources.
Long-Range Attack

Mainly a PoS concern: an attacker acquires old private keys that once held a large stake and rebuilds an alternate chain from far in the past, since old stake is now cheap/worthless to acquire.

Mitigation: checkpointing and "weak subjectivity" — new nodes trust a recent, socially-agreed checkpoint rather than replaying from genesis.
Nothing-at-Stake

Early PoS concern: because voting costs almost nothing, validators could cheaply vote on every competing fork simultaneously, weakening the guarantee that only one history wins.

Mitigation: slashing — validators lose real staked funds for signing conflicting blocks, making forking costly again.
Selfish Mining

A PoW miner secretly withholds a found block and keeps mining a private lead, releasing blocks strategically to waste honest miners' effort and earn disproportionate rewards.

Mitigation: propagation-timing rules and tie-breaking policies that reduce the payoff of withholding blocks.
Immutability & transparency, revisited
  • Immutability is only as strong as the cost of the attack needed to break it — a "51% attack" is, literally, the price tag on rewriting history.
  • Transparency (every node holding the full ledger) is what makes these attacks detectable even when they can't be fully prevented — forks and double-spends are publicly visible the moment they happen.
  • Permissioned algorithms (PBFT, PoA) trade Sybil-resistance-by-cost for Sybil-resistance-by-identity-vetting — cheaper, but reliant on trusting the vetting process.

        Block 07 · Layer 1, Layer 2 & Layer 3

        Base chains, scaling chains, and application chains

        Definition

        Layer 1 (L1) is a base blockchain that runs its own consensus algorithm and finalises transactions directly on its own ledger (e.g. Bitcoin, Ethereum, Solana). Layer 2 (L2) is a protocol built on top of an L1 that processes transactions off the base chain for speed and low cost, then periodically settles a compressed proof or batch back down to the L1 — inheriting its security instead of running an independent trust model from scratch. Layer 3 (L3) is a further application-specific layer built on top of an L2 (e.g. a game or app-chain settling to an L2 rollup), trading a little more shared security for even cheaper, more customised execution — sometimes called an "app-chain" or "hyperchain."

        How a Layer 2 relies on its Layer 1

        01Users transact on the L2 (fast, cheap, off-chain execution)
        →
        02L2 batches thousands of transactions together
        →
        03A compressed proof / transaction batch is posted to the L1
        →
        04L1's own consensus algorithm validates and finalises it
        →
        05L2 state is now as secure as the underlying L1

        Layer 1 — base chains

        Run independent consensus. Security, decentralisation and finality happen here.

        Bitcoin — PoW Ethereum — PoS Solana — PoH+PoS Cardano — Ouroboros PoS EOS — DPoS VeChain — PoA

        Layer 2 — scaling solutions

        No independent security model of their own for trust — they borrow it from an L1.

        Lightning Network → secures via Bitcoin Arbitrum (Optimistic Rollup) → secures via Ethereum Optimism (Optimistic Rollup) → secures via Ethereum zkSync (ZK-Rollup) → secures via Ethereum Polygon PoS → secures via Ethereum
        Key points
        • L1 changes are hard and slow (consensus-level upgrades affect the whole network); L2s can iterate quickly since they sit above the trust-critical layer.
        • Rollups (Optimistic & ZK) differ in how they prove correctness: Optimistic rollups assume transactions are valid unless challenged (fraud proofs); ZK-rollups mathematically prove validity upfront (validity proofs).
        • An L2's consensus mechanism (if it even has one, e.g. a sequencer set) is a separate, usually more centralised, layer that only matters for liveness — final security still comes from the L1.

        Scaling techniques, compared

        "Layer 2" is a category — these are the actual techniques used to build one.

        Optimistic Rollups

        Execute transactions off-chain, post the compressed data on-chain, and assume it's valid unless someone submits a fraud proof within a challenge window (often ~7 days for full withdrawal finality).

        Arbitrum · Optimism · Base

        ZK-Rollups

        Execute off-chain and post a succinct cryptographic validity proof (a zk-SNARK/STARK) alongside the data — the L1 verifies the proof directly, so there's no challenge-period delay.

        zkSync · Starknet · Polygon zkEVM

        Plasma

        An earlier scaling design: child chains process transactions and periodically commit a Merkle root to the L1. Users can "exit" back to the L1 with a fraud proof if the child chain misbehaves — cheaper than rollups but weaker for arbitrary smart-contract logic.

        OMG Network · early Polygon

        Sidechains

        Independent blockchains connected to an L1 via a two-way bridge, but running their own consensus and validator set — faster and cheaper, but not inheriting L1 security directly (a bridge hack affects the sidechain only).

        Polygon PoS · Ronin

        State Channels

        Two or more parties lock funds in an on-chain contract, then transact unlimited times off-chain by exchanging signed updates directly with each other, only settling the final state on-chain when the channel closes.

        Bitcoin Lightning Network · Raiden (Ethereum)

        Layer 3 / App-chains

        A dedicated chain for a single application, built on top of an L2 rollup rather than directly on L1 — shares the L2's proving/settlement infrastructure while getting full control over its own execution environment and fees.

        Arbitrum Orbit chains · zkSync Hyperchains

        Block 09 · Case studies

        When the trade-off actually mattered

        Four moments where a consensus algorithm's design directly caused a real, observable outcome — good or bad.

        Bitcoin — the 2010 value-overflow incident

        PoW · Immutability under stress

        A bug let someone create 184 billion BTC out of thin air in one transaction. Developers patched the client and the community adopted a corrected chain within hours — a soft fork, not a rewrite of settled history.

        Shows the difference between fixing a bug going forward and actually breaking immutability: no already-confirmed, legitimate transaction was reversed.

        The DAO hack & the Ethereum/Classic split

        PoW (at the time) · Governance vs. immutability

        An attacker drained ~3.6M ETH from a smart contract in 2016. The community controversially voted to hard-fork and effectively reverse the theft — a minority rejected this, continuing the original chain as Ethereum Classic.

        A live demonstration that "immutable" is a social commitment enforced by node operators, not a law of physics.

        Solana's 2021–2022 network outages

        PoH + PoS · Scalability vs. resilience

        Solana suffered several multi-hour full outages when transaction spam (e.g. bot activity during NFT mints) overwhelmed validator memory pools, since its design assumes very high, reliable throughput.

        Illustrates the scalability side of the trilemma cutting the other way: optimising hard for speed can leave less margin for degraded conditions.

        Ethereum's 2022 Merge

        PoW → PoS migration

        Ethereum switched its entire consensus layer from PoW to PoS in a single, live event, cutting network energy use by roughly 99.95% without a single day of downtime or a lost block.

        The largest live consensus-algorithm migration ever performed on a major public chain — proof that even the "immutable" mechanism itself can evolve, given years of preparation.

        Block 10 · Interview questions & FAQ

        Questions worth being able to answer

        Common interview and exam questions on this topic — click to reveal a model answer.

        Why can't a blockchain just use a simple majority vote among all nodes?+
        Because anyone can spin up unlimited free node identities in a permissionless network (a Sybil attack) — a raw one-node-one-vote system would let an attacker outvote everyone by creating enough fake nodes. Consensus algorithms tie voting power to something scarce and costly instead: computation (PoW), capital at risk (PoS), or vetted identity (PoA/PBFT).
        What's the actual difference between PoW and PoS security?+
        PoW security is rooted in external, physical cost — attacking it means outspending the world on hardware and electricity, and that cost is sunk whether you attack or not. PoS security is rooted in internal, economic cost — an attacker must own a huge amount of the very asset they're attacking, and dishonest behaviour causes that stake to be programmatically destroyed (slashed).
        Can two Layer 2s on the same Layer 1 talk to each other directly?+
        Not natively — most L2s only have a direct trust relationship with their underlying L1, not with each other. Moving assets or messages between two L2s (say, Arbitrum and Optimism) typically requires going through the L1 or using a third-party cross-rollup bridge, which reintroduces some trust assumptions.
        Why is DPoS considered less decentralised than PoS?+
        Pure PoS allows (in principle) any staker to become a validator. DPoS deliberately shrinks the validator set to a small, fixed number of elected delegates (e.g. 21) to reduce coordination overhead and boost speed — but that means far fewer entities hold real block-producing power, concentrating trust.
        What does "finality" mean, and why do algorithms differ on it?+
        Finality is the guarantee that a confirmed block cannot later be reversed. PoW only offers probabilistic finality — a block gets safer the more blocks are mined on top of it, but is never mathematically 100% final. BFT-style algorithms (PBFT, and PoS chains with BFT-style finality gadgets) offer absolute finality — once 2/3+ of validators commit, the block is instantly and provably irreversible.
        Is a Layer 2 as secure as its Layer 1?+
        It depends on the design. ZK-rollups and optimistic rollups aim to inherit full L1 security because the L1 itself verifies a proof or fraud-challenge of every state transition. Sidechains do not — they run independent validator sets, so compromising the sidechain's own consensus (not the L1's) is enough to steal funds, as seen in bridge hacks like Ronin.
        Block 11 · Glossary

        Terms worth knowing cold

        NodeAny computer running the blockchain's software and holding a copy of the ledger.
        Validator / MinerA node that actively participates in proposing or confirming new blocks.
        FinalityThe point at which a confirmed block is guaranteed (probabilistically or absolutely) never to be reversed.
        ForkA divergence in the blockchain's history, either accidental (temporary) or a deliberate protocol change (hard/soft fork).
        SlashingDestroying part of a validator's staked funds as punishment for provably dishonest behaviour.
        GasThe fee unit paid to compensate validators for the computational cost of executing a transaction or smart contract.
        Smart contractSelf-executing code deployed on a blockchain that runs exactly as programmed once triggered.
        BridgeInfrastructure that lets assets or data move between two otherwise-separate blockchains.
        Double-spendSpending the same digital coin twice — the core problem consensus algorithms exist to prevent.
        TPSTransactions per second — the primary throughput metric used to compare scalability.
        MempoolThe pool of pending, unconfirmed transactions a node holds before they're included in a block.
        ShardingSplitting a network's data/validation load across multiple parallel chains ("shards") to scale horizontally.
